# Development Context

## 1. 项目目标

根据 `README.md`，这是一个面向 NeoForge 1.21.1 的生态演替模组，核心目标如下：

- 以区块（16×16）为最小演替单位。
- 根据群系、温度、湿度和 JSON 配置决定演替目标。
- 在演替完成时真实修改区块群系，而不是只做视觉模拟。
- 用野生植物的数量、质量和寿命驱动演替进度。
- 允许在缺少外部模组时自动降级，在检测到 `Dynamic Trees` 时再启用联动。

## 3. 明确约束

### 3.1 信息来源约束

- 当前开发只允许参考 `README.md` 和当前分支工作区内容。
- 不查看、不比对、不引用其他 Git 分支内容。
- 如果需求细节不足，应在当前分支文档里记录为待确认项，而不是去其他分支找答案。

### 3.2 实现层面的硬约束

- 演替是低频计算，必须优先控制性能开销。
- 区块卸载后不继续进行演替评估。
- 植物管理对象只包含野生植物，不包含农作物。
- 数据需要持久化到区块级对象，README 明确建议使用 NeoForge `Data Attachments`。
- 演替路径和植物列表必须由数据驱动配置加载。

## 4. 已知差异与待确认事项

### 4.1 命名决议

- 已确认统一命名使用 `Ecoflux`，并已完成基础工程命名切换。
- 后续新增代码、资源、数据文件统一使用 `ecoflux` 命名空间。
- 工作区里如果仍存在 `succession` 遗留文件，应视为历史残留，后续按功能迁移或移除，不再作为新实现的命名依据。

### 4.2 当前缺口

- 缺少 JSON 数据格式定义与加载器。
- 缺少植物追踪、生命周期监听和点数评估机制。
- 缺少群系替换实现。
- 缺少网络同步和兼容层骨架。
- 当前附件数据结构仍是骨架，尚未接入真实区块初始化和运行流程。

## 5. 当前建议的开发节奏

建议按下面的顺序进入正式开发：

1. 在现有附件骨架之上接入真实区块初始化与保存时机。
2. 实现数据驱动配置加载。
3. 实现植物跟踪、队列与进度系统。
4. 处理群系替换。
5. 最后补同步、调试工具和可选兼容。

## 6. 开发时优先关注的风险

- 群系替换 API 的可行性与粒度是否符合 1.21.1 版本能力。
- 区块级活跃植物映射如何避免在大地图中失控增长。
- 植物死亡/破坏监听是否覆盖自然凋零、玩家破坏和外部模组行为。
- 数据驱动配置如何同时支持“单一路径”和“同源多分支路径匹配”。
