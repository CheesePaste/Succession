# Succession Roadmap 与规范 Spec

## 1. 文档目标

- 模组名称：`Succession`
- Minecraft 版本：`1.21.1`
- 模组平台：`NeoForge`
- 文档状态：`v0.1 草案`
- 文档用途：作为玩法设计、技术实现、数据驱动格式、测试验收与版本推进的统一基线

本规范定义 `Succession` 的核心目标：在**不新增自定义群系类型**的前提下，基于原版群系参数与世界动态状态，实现原版群系之间的**渐进式生态演替**、**地物逐步变化**以及**边界合理扩张/退化**。

---

## 2. 产品愿景

`Succession` 的目标不是替换原版世界生成，而是让已经生成出来的世界在玩家游玩过程中继续“活着”：

- 平原在湿润、植被增加的条件下，逐渐出现灌木、稀疏乔木，最终演化为森林。
- 干旱、扰动或缺水条件会让原本较繁茂的区域退化，甚至走向沙漠化。
- 演替不是瞬时切换，而是先发生地物层面的量变，再触发生物群系身份的质变。
- 顶级或退化型群系可以向周边满足条件的区域扩张，但必须遵守地理合理性与性能预算。

---

## 3. 设计原则

### 3.1 群系身份原则

1. 只使用原版或其他模组已经存在的群系 ID。
2. 不新增“过渡群系”“中间群系”“隐藏群系”。
3. 演替的中间态通过**进度状态**与**地物配置**表现，而不是通过新增 biome type 表现。

### 3.2 渐进原则

1. 每个生态单元都存在 `0.0 ~ 1.0` 的演替进度。
2. 演替进度先影响植被、树木密度、地表装饰等表现。
3. 只有当进度达到阈值并持续稳定一段时间，才允许群系身份切换。

### 3.3 多路径原则

1. 同一来源群系允许存在多个候选目标群系。
2. 候选路径由环境条件、邻域影响、局部植被状态共同竞争决定。
3. 允许正向演替、逆向退化、寒冷偏移、干旱偏移等不同方向。

### 3.4 数据驱动原则

1. 演替路径、阈值、权重、时间尺度、地物阶段均通过 JSON 配置定义。
2. 代码只负责读取规则、计算状态、调度更新与应用结果。
3. 整合包应能增删演替路径、覆盖参数、禁用特定转化。

### 3.5 性能优先原则

1. 只处理已加载区块。
2. 所有重计算任务拆分并限额执行。
3. 长周期统计、区域识别、边界扩张必须支持异步分析与主线程提交。

### 3.6 地理合理性原则

1. 海洋、河流、下界、末地等特殊群系默认不参与普通陆地演替。
2. 扩张不能跨越明显不兼容的群系类别。
3. 演替与扩张必须参考原版温度、湿度、降水、邻域群系和地表状态。

## 4. 术语定义

### 4.1 生态采样单元 `Ecology Sample Cell`

- 定义：以地表为中心的 `4x4` 水平采样单元，作为最小演替判断粒度。
- 设计原因：贴近原版 biome 存储的水平粒度，精度高于整区块，同时仍可控。

### 4.2 工作区块 `Work Chunk`

- 定义：玩家已加载的 `16x16` 区块，是调度、持久化和任务分发的基本单位。
- 一个 `Work Chunk` 内包含多个 `Ecology Sample Cell`。

### 4.3 生态区域 `Ecology Region`

- 定义：在当前已加载范围内，由相邻且“当前主群系一致”的采样单元构成的连通区域。
- 用途：用于保证演替判断不只看单点，而是考虑连续区域的完整性。

### 4.4 演替路径 `Succession Path`

- 定义：从一个来源群系到一个目标群系的规则记录。
- 例：`minecraft:plains -> minecraft:forest`

### 4.5 演替阶段 `Succession Stage`

- `Dormant`：无明显变化，维持原群系。
- `Accumulating`：同群系内量变，逐步积累。
- `ThresholdReady`：满足切换阈值，等待稳定确认。
- `Converted`：已切换到目标群系，开始新一轮稳定期。
- `StablePeak`：顶级群落或退化极端群落，默认只响应外部扰动或边界扩张。

### 4.6 扰动 `Disturbance`

- 会阻碍或逆转演替的事件与条件，例如：
  - 玩家大量砍树
  - 火灾
  - 爆炸
  - 长期低植被覆盖
  - 极端温湿度偏移

## 5. 范围与非目标

### 5.1 V1 范围

- 原版地表群系之间的演替与退化
- 同一群系内部地物表现的分阶段变化
- 已加载区块内的动态更新与持久化
- 数据包可配置的演替图谱与展示阶段
- 顶级群系/低级群系的边界扩张或侵蚀

### 5.2 非目标

- 不重做原版世界生成器
- 不在 V1 中实现季节系统
- 不在 V1 中做全世界离线扫描
- 不在 V1 中支持所有维度的复杂生态逻辑
- 不在 V1 中模拟真实土壤化学、地下水或动物链条

## 6. 核心玩法 Spec

### 6.1 演替图谱模型

演替不是单线树，而是有向图：

- 一个来源群系可有多个候选目标
- 每条边定义自己的环境条件、速率、门槛与展示配置
- 路径分为三类：
  - `progressive`：正向演替，如 `plains -> forest`
  - `regressive`：逆向退化，如 `plains -> desert`
  - `lateral`：侧向偏移，如 `plains -> taiga` 或 `forest -> old_growth_birch_forest`

### 6.1.1 示例图谱

- `plains -> forest`
- `forest -> dark_forest`
- `forest -> old_growth_birch_forest`
- `plains -> taiga`
- `taiga -> snowy_plains`
- `plains -> desert`
- `forest -> plains`

说明：

- 这些示例不是硬编码清单，而是数据包中可定义的候选边。
- 最终生效路径由候选评分与阈值竞争决定。

### 6.2 驱动输入

每个采样单元在更新时读取以下输入：

### 6.2.1 静态输入

- 当前群系 ID
- 原版温度
- 原版湿度 / downfall
- 原版降水类型
- 海拔区间
- 所在维度
- 地表方块类别

### 6.2.2 动态输入

- 树木密度
- 木本植物覆盖度
- 草本植物覆盖度
- 枯木 / 蘑菇 / 林下装饰计数
- 裸地比例
- 周边相邻采样单元群系统计
- 扰动计数器
- 最近稳定时长

### 6.3 指标定义

所有动态指标都标准化到 `0.0 ~ 1.0`。

### 6.3.1 树木密度 `tree_density`

- 统计半径内树干方块、树叶方块与可识别树冠结构
- 对大树和密林给予更高权重
- 用于判断平原向森林、森林向深色森林等方向的可能性

### 6.3.2 植被覆盖度 `vegetation_cover`

- 统计草、高草、灌木、花、苔藓、地衣等覆盖型方块
- 用于判断生态恢复程度

### 6.3.3 裸地比例 `bare_ground_ratio`

- 统计泥土、沙、石头、裸露地表比例
- 比例越高，越倾向退化或阻碍正向演替

### 6.3.4 邻域影响 `neighborhood_pressure`

- 统计相邻采样单元或相邻区域的群系分布
- 如果某目标群系已在边界广泛存在，则对对应路径给予正权重

### 6.3.5 扰动压力 `disturbance_pressure`

- 由火焰、爆炸、砍伐、方块频繁更改等事件累积
- 该值衰减缓慢，用于防止区域被瞬时修复

### 6.4 路径评分规则

每个来源群系的所有候选路径在每次评估时计算一个 `candidate_score`：

`candidate_score = base_weight * environment_match * neighborhood_pressure * vegetation_factor * disturbance_modifier`

约束如下：

1. 所有因子归一到 `0.0 ~ 1.0` 或有限倍数范围。
2. 只有 `candidate_score >= activation_threshold` 的路径才进入激活状态。
3. 若多个候选同时满足，选择得分最高者作为当前目标路径。
4. 若新路径要替换旧路径，需超过旧路径分数一定滞回边距，默认建议 `15%`。

### 6.4.1 滞回机制

为避免目标路径反复横跳：

- 候选目标切换需要连续满足条件若干次采样窗口
- 默认要求连续稳定 `2` 个游戏日
- 切换后进入 `cooldown`，短时间内不再重新选路

### 6.5 演替进度规则

每个采样单元都维护：

- `current_biome`
- `target_biome`
- `active_path_id`
- `progress`
- `stage`
- `stability_ticks`
- `last_transition_time`

### 6.5.1 进度增长

`progress_delta = base_rate * candidate_score * time_factor * seasonal_like_modifier * disturbance_modifier`

其中：

- `base_rate`：路径基础速率
- `candidate_score`：当前目标路径评分
- `time_factor`：服务器 tick 预算与采样间隔修正
- `seasonal_like_modifier`：为未来季节系统预留，V1 默认为 `1.0`
- `disturbance_modifier`：扰动越大，增长越慢，甚至为负

### 6.5.2 阶段阈值

建议默认阈值：

- `0.00 ~ 0.24`：初期变化
- `0.25 ~ 0.59`：中期变化
- `0.60 ~ 0.89`：后期变化
- `0.90 ~ 0.99`：临界准备
- `>= 1.00`：允许切换群系身份

### 6.5.3 切换条件

只有在以下条件同时满足时，才将 `current_biome` 变更为 `target_biome`：

1. `progress >= 1.0`
2. 目标路径连续稳定满足阈值
3. 当前采样单元不处于保护类别
4. 目标群系与所在维度、地表类型兼容
5. 当前任务预算允许提交 biome 写入

切换完成后：

- `progress` 重置为 `0`
- `target_biome` 清空
- `stage` 置为 `Converted`
- 开始新群系的稳定观察期

### 6.6 同群系内的渐进表现

在正式切换群系前，玩家应先看到地物层面的连续变化。

### 6.6.1 表现层原则

1. 当前 biome ID 不变，但地表地物生成/补种/衰退表现变化。
2. 表现由“展示配置 `presentation_profile` + progress 区间”驱动。
3. 同一群系可因不同目标路径而呈现不同中间态。

### 6.6.2 示例：`plains` 向 `forest`

- `0% ~ 20%`：少量灌木、零星橡树、草较稀疏
- `20% ~ 60%`：橡树与桦树混合增加，高草增多
- `60% ~ 90%`：林下装饰出现，树木密度明显上升
- `90% ~ 100%`：接近林地观感，等待群系身份切换

### 6.6.3 示例：`plains` 向 `desert`

- `0% ~ 20%`：草减少，裸土增加
- `20% ~ 60%`：枯死灌木、小片沙化
- `60% ~ 90%`：植被显著减少，地表偏干黄
- `90% ~ 100%`：接近干旱地貌，等待切换

### 6.7 顶级群系与退化群系的外部扩张

### 6.7.1 触发条件

当群系位于其演替图谱的终点，且持续稳定达到设定天数后，可向边界邻域扩张：

- 顶级群系示例：`dark_forest`、`old_growth_birch_forest`
- 退化群系示例：`desert`、`badlands`

### 6.7.2 扩张规则

1. 只能从边界采样单元向直接相邻单元尝试扩张。
2. 目标单元必须满足目标群系的基础环境阈值。
3. 默认禁止跨越下列类别：
   - 海洋与深海
   - 河流
   - 下界与末地
   - 蘑菇岛
   - 明显不兼容的高山/洞穴专属群系
4. 扩张速率显著慢于内部演替速率。
5. 扩张优先影响与自身环境接近的边界单元。

### 6.7.3 侵蚀规则

低级或退化群系可进行“逆向扩张”，但必须更保守：

- 需要持续满足干旱、低覆盖、高裸地等条件
- 需要更长的稳定时间
- 若玩家修复环境，应允许较慢逆转

### 6.8 区域完整性处理

为避免只更新一半区域导致逻辑割裂：

1. 当某加载区块中的采样单元被判定为“活跃演替”，系统会尝试识别其所在的已加载连续区域。
2. 该连续区域中的相关采样单元被标记为 `related_active_region`。
3. 仅对已加载范围执行真实更新；未加载部分不强制加载世界。
4. 若区域在边界处被截断，则对边界单元打上 `incomplete_observation` 标记，降低激进决策概率。

这样既满足“只在加载区块工作”，又避免边缘判断失真。

## 7. 技术架构 Spec

### 7.1 模块划分

建议划分为以下逻辑模块：

1. `Rule Registry`
   - 加载演替路径、展示配置、例外规则
2. `Ecology Sampler`
   - 从已加载区块提取温湿度、树木密度、覆盖度等指标
3. `Region Tracker`
   - 识别和维护加载范围内的连续生态区域
4. `Succession Evaluator`
   - 为采样单元计算候选路径、得分、目标与进度
5. `Presentation Applicator`
   - 依据阶段变化补种、替换、移除地表装饰
6. `Biome Transition Applicator`
   - 在满足条件时提交 biome 身份变更
7. `Persistence Layer`
   - 持久化采样单元状态、区域状态与冷却信息
8. `Task Scheduler`
   - 控制每 tick / 每秒预算，拆分异步与主线程任务

### 7.2 数据生命周期

### 7.2.1 加载时

1. 世界/区块加载
2. 读取已保存的演替状态
3. 为新增区块初始化默认状态
4. 将其加入待采样队列

### 7.2.2 周期更新

1. 抽取部分活跃区块
2. 重建或局部更新区域连通信息
3. 重新采样局部动态指标
4. 计算候选路径得分
5. 更新 `progress` 与 `stage`
6. 如满足条件，则提交地物变化或 biome 写入
7. 按批次保存脏数据

### 7.2.3 卸载时

1. 刷新脏状态到区块级持久化
2. 从调度器移除运行时缓存

### 7.3 持久化模型

### 7.3.1 区块级状态

每个 `Work Chunk` 至少需要保存：

- 版本号
- 上次采样游戏时间
- 本区块采样单元列表
- 活跃路径缓存
- 扰动计数器
- 脏标记

### 7.3.2 采样单元状态

建议字段：

```json
{
  "cell_pos": "x,z",
  "current_biome": "minecraft:plains",
  "target_biome": "minecraft:forest",
  "active_path": "succession:plains_to_forest",
  "progress": 0.64,
  "stage": "accumulating",
  "stability_ticks": 4200,
  "disturbance": 0.15,
  "last_transition_time": 1234567,
  "flags": [
    "related_active_region"
  ]
}
```

### 7.3.3 世界级状态

需要世界级记录：

- 数据版本
- 规则版本哈希
- 全局默认参数
- 已知区域缓存索引（可选）
- 黑名单群系类别

### 7.4 调度与性能预算

### 7.4.1 更新频率

建议分层更新：

- 高频：活跃边界区块，`20 ~ 60` 秒一次
- 中频：普通已加载区块，`2 ~ 5` 分钟一次
- 低频：稳定区块，`5 ~ 15` 分钟一次

### 7.4.2 Tick 预算

建议配置：

- 每 tick 最多处理固定数量区块
- 每次采样最多扫描固定数量采样单元
- 超预算时把剩余任务重新排队

### 7.4.3 异步边界

可异步：

- 候选路径评分
- 区域连通图计算
- 指标汇总与统计

必须主线程提交：

- 方块实际放置/移除
- biome 写入
- 区块持久化对象变更

### 7.5 区块加载策略

### 7.5.1 原则

1. 不主动强制加载远处区块。
2. 只在玩家或服务器已加载范围内工作。
3. 对跨区块生态区域只做“已加载子集上的完整性近似”。

### 7.5.2 相关加载区定义

当某区域进入活跃演替：

- 将其所属已加载连续区域标记为“演替相关加载区”
- 调度器优先处理该区域
- 若边界区块卸载，则区域状态切为“部分观测”

### 7.6 地物表现实现策略

V1 建议采用混合方案：

1. 对“可渐进补充”的内容使用运行时补种/补放：
   - 树木
   - 花草
   - 蘑菇
   - 枯木
2. 对“生物群系身份变化后的长期表现”依赖 biome 切换后的后续自然生成与随机刻联动。
3. 不在 V1 中大规模重写完整世界生成管线。

### 7.6.1 表现层约束

- 不能在短时间内大量刷树造成性能尖峰
- 每次更新只允许少量增删
- 生成位置必须通过可存活性检测
- 不破坏玩家放置的重要方块

### 7.7 兼容性策略

### 7.7.1 默认支持

- 原版主世界地表群系
- 其他模组 biome 只要具有合法 biome ID，即可通过数据包接入

### 7.7.2 黑名单类别

默认建议黑名单：

- `minecraft:ocean`
- `minecraft:deep_ocean`
- `minecraft:river`
- `minecraft:frozen_ocean`
- `minecraft:the_void`
- 下界与末地全体群系

### 7.7.3 数据缺失回退

如果某群系没有配置演替路径：

- 默认只采样不演替
- 不报致命错误
- 在日志中发出一次性警告

## 8. 数据驱动格式 Spec

### 8.1 演替路径文件

建议目录：

`data/<namespace>/succession_paths/*.json`

建议结构：

```json
{
```

## 10. 验收标准

### 10.1 功能验收

- 平原在湿润条件下可逐步转向森林系路径
- 平原在干旱条件下可逐步转向沙漠系路径
- 同一来源群系在不同环境下可选择不同目标
- 顶级群系可缓慢扩张到合理邻域
- 退化群系可在长期恶化条件下侵蚀邻域

### 10.2 表现验收

- 玩家在群系切换前能观察到多阶段地物变化
- 不同路径的中间态外观有可辨识差异
- 切换不会产生过度突兀的大面积瞬变

### 10.3 技术验收

- 只在加载区块内计算
- 区块卸载/重载后状态不丢失
- 数据包错误不会导致世界损坏
- 日志足以定位规则冲突与异常状态

### 10.4 性能验收

- 调度系统具备硬性预算上限
- 大量加载区块时任务可被分帧而非集中爆发
- 扩张、演替、采样不会在单 tick 内集中执行

## 12. 风险与决策记录

### 12.1 主要风险

1. biome 实际写入与已有世界内容之间可能存在视觉割裂。
2. 地物渐进补种如果过于激进，会产生性能波峰。
3. 多路径竞争容易导致规则作者配置出互相冲突的循环。
4. 只处理加载区块会使演替在玩家不在场时暂停，这是设计选择，不是 bug。

### 12.2 对应策略

1. 通过中间表现阶段降低切换突兀感。
2. 通过硬预算和分帧避免批量刷地物。
3. 通过校验器检测明显循环、冲突阈值与不可达路径。
4. 在文档中明确“演替是观察驱动的世界活动”，仅在已加载区块推进。

## 13. V1 默认建议规则集

首发建议优先支持以下路径，方便形成最小可玩闭环：

1. `minecraft:plains -> minecraft:forest`
2. `minecraft:forest -> minecraft:dark_forest`
3. `minecraft:plains -> minecraft:desert`
4. `minecraft:plains -> minecraft:taiga`
5. `minecraft:taiga -> minecraft:snowy_plains`
6. `minecraft:forest -> minecraft:plains`

这样可以同时覆盖：

- 正向增植演替
- 干旱退化
- 寒冷偏移
- 顶级群落稳定

## 15. 实施结论

`Succession` 的 V1 应围绕一个核心闭环展开：

1. 在已加载区块中采样生态指标。
2. 以数据驱动规则为每个采样单元选择演替方向。
3. 先通过地物层面的渐进变化展示过程。
4. 在稳定阈值满足后切换为新的原版群系。
5. 对终点群系进行受约束的缓慢外扩。

只要这个闭环稳定成立，后续无论是扩展更多群系、接入其他模组 biome，还是增加季节与灾害系统，都可以在同一套架构上继续演进。
